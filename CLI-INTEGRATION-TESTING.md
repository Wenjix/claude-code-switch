# CLI Integration Testing Guide

**Author:** Manus AI  
**Last Updated:** November 13, 2025

## Overview

The CLI integration test suite validates the complete workflow from the CCM Dashboard to command-line execution. These tests ensure that configurations generated by the web interface are correctly written to disk, properly formatted for shell consumption, and accurately represent the user's model selection. With 25 comprehensive tests achieving 100% pass rate, the suite provides confidence that the dashboard and CLI tools work together seamlessly.

## Testing Philosophy

CLI integration testing differs fundamentally from unit testing because it validates the interaction between multiple system components rather than isolated functions. The test suite focuses on the critical handoff point where the dashboard writes configuration data that the CLI tools must read and apply.

### Why Integration Testing Matters

The CCM Dashboard and CLI tools form a distributed system where data flows across process boundaries and file system operations. Unit tests cannot catch integration failures such as file format mismatches, permission issues, or race conditions between write and read operations. Integration tests validate that these components work together correctly in realistic scenarios.

**File format compatibility** represents a critical integration point. The dashboard writes JSON configuration files that the CLI tools must parse. If the dashboard changes its output format without corresponding updates to the CLI parser, the system breaks. Integration tests catch these incompatibilities by validating the actual file format rather than mocking file operations.

**Environment variable handling** requires careful coordination between the dashboard and shell environment. The dashboard generates export statements that must follow shell syntax rules, handle special characters correctly, and preserve variable substitution patterns like `$DEEPSEEK_API_KEY`. Integration tests verify that generated scripts execute correctly in real shell environments.

**Provider-specific configurations** add complexity because different AI providers require different base URLs, model identifiers, and API key patterns. Integration tests validate that each provider's configuration is correctly formatted and contains all required fields.

## Test Architecture

The CLI integration test suite uses a temporary directory approach that isolates each test from the file system and prevents interference between tests.

### Temporary Test Environment

Each test runs in its own temporary directory created with Node.js's `fs.mkdtemp` function. This directory serves as a sandboxed HOME directory where configuration files can be written and read without affecting the real user environment or other tests.

The test setup creates a temporary directory before each test and stores its path in the `testDir` variable. The `configFilePath` variable points to `.ccm_active_model` within this temporary directory, simulating the location where the dashboard writes configuration files in production.

After each test completes, the cleanup code removes the temporary directory and all its contents, ensuring no test artifacts remain on the file system. This cleanup happens even if the test fails, preventing accumulation of temporary files over time.

### Environment Variable Isolation

Tests temporarily override the `HOME` environment variable to point to the test directory. This ensures that any code reading `~/.ccm_active_model` actually reads from the test directory rather than the user's real home directory.

The original `HOME` value is saved before each test and restored afterward, preventing tests from affecting the environment of subsequent tests or the test runner itself. This isolation is critical for test reliability and reproducibility.

## Test Coverage

The CLI integration test suite includes 25 tests organized into 10 logical groups, each validating a specific aspect of the dashboard-to-CLI workflow.

### Config File Reading Tests

The most fundamental integration point is the dashboard's ability to write configuration files that the CLI can read. Five tests validate this capability across different scenarios.

**Valid config file reading** tests verify that the CLI can successfully parse configuration files generated by the dashboard. The test creates a realistic configuration for DeepSeek including provider identifier, timestamp, and environment variable mappings. After writing this configuration to the test file, the test reads it back and validates that all fields are preserved correctly.

**Claude Sonnet configuration** tests ensure that configurations for Anthropic's official Claude models are handled correctly. These configurations differ from third-party providers because they don't require a custom base URL—the ANTHROPIC_BASE_URL field is set to an empty string. The test validates that empty strings are preserved rather than being converted to null or undefined.

**KIMI configuration** tests validate handling of the Moonshot AI provider, which requires a specific base URL and model identifier format. This test ensures that provider-specific URLs and model names are correctly preserved through the write-read cycle.

**Invalid JSON handling** tests verify that the system fails gracefully when encountering corrupted configuration files. Rather than crashing or producing undefined behavior, the code should throw a clear error that can be caught and handled appropriately.

**Missing file handling** tests ensure that attempts to read non-existent configuration files produce appropriate errors rather than undefined behavior. This validates that the system can detect when users haven't yet configured a model through the dashboard.

### Config File Writing Tests

Three tests validate that the dashboard correctly writes configuration files in the expected format.

**JSON format validation** ensures that written files contain valid JSON that can be parsed by standard JSON parsers. The test writes a configuration, reads it back, and verifies that the parsed object matches the original data structure.

**Field preservation** tests validate that all configuration fields are written to disk without loss or corruption. This includes not just the required fields but also optional metadata fields that provide context about when and how the configuration was created.

**Metadata handling** tests verify that additional metadata fields beyond the core configuration are preserved. This allows the dashboard to include version information, source tracking, and other contextual data that might be useful for debugging or analytics.

### Environment Variable Format Tests

Three tests validate that configuration data is correctly formatted for shell consumption.

**Bash export statement generation** tests verify that environment variable assignments follow proper bash syntax. Each variable should be exported with the format `export VAR_NAME="value"`, with values properly quoted to handle spaces and special characters.

**Empty value handling** tests ensure that empty strings are correctly represented in export statements. An empty ANTHROPIC_BASE_URL should generate `export ANTHROPIC_BASE_URL=""` rather than being omitted or set to null.

**Special character escaping** tests validate that values containing quotes, ampersands, or other shell metacharacters are properly escaped to prevent syntax errors or security vulnerabilities. For example, a value containing double quotes should escape them as `\"` to prevent premature string termination.

### Shell Script Generation Tests

Two tests validate the generation of complete shell scripts from configuration data.

**Valid bash script generation** tests verify that the complete script includes a proper shebang line (`#!/bin/bash`), comments documenting the provider and timestamp, and export statements for all environment variables. The generated script should be executable and produce the intended environment changes when sourced.

**Line ending consistency** tests ensure that scripts use Unix-style line endings (`\n`) consistently. Mixed line endings can cause issues when scripts are executed on Unix-like systems, so maintaining consistency is important for reliability.

### Config Validation Tests

Three tests validate that configuration data includes all required fields and follows expected formats.

**Required field validation** tests verify that configurations include provider, timestamp, and config object fields. Missing any of these fields should be detectable so the system can reject invalid configurations.

**Missing field detection** tests validate that the system can identify configurations missing required fields. This prevents incomplete configurations from being accepted and potentially causing runtime errors.

**Provider value validation** tests ensure that provider identifiers match the list of supported providers. This prevents typos or invalid provider names from being accepted.

**Timestamp format validation** tests verify that timestamps follow ISO 8601 format and can be parsed as valid dates. This ensures consistency in timestamp handling across the system.

### File Permission Tests

One test validates that configuration files are created with appropriate permissions.

**Readable permission validation** ensures that written configuration files can be read by the user who created them. While this test doesn't validate specific permission bits (which vary by system), it confirms that the file is accessible and readable.

### Error Handling Tests

Three tests validate that the system handles various error conditions gracefully.

**Write error handling** tests verify that attempts to write to invalid paths (such as directories the user doesn't have permission to access) produce appropriate errors rather than crashing or hanging.

**Corrupted file handling** tests ensure that malformed JSON in configuration files is detected and reported clearly. This prevents the system from attempting to use invalid configuration data.

**Empty file handling** tests validate that empty configuration files are rejected rather than being treated as valid (but empty) configurations. An empty file likely indicates a write failure or corruption rather than a legitimate empty configuration.

### Timestamp Handling Tests

Two tests validate timestamp preservation and format handling.

**Timestamp precision preservation** tests ensure that timestamps maintain their full precision through write-read cycles. ISO 8601 timestamps include milliseconds, and this precision should be preserved rather than being truncated to seconds.

**Multiple format support** tests verify that the system can handle timestamps in different formats including ISO 8601 strings, UTC strings, and numeric Unix timestamps. This flexibility ensures compatibility with different timestamp sources.

### Provider-Specific Configuration Tests

Three tests validate that configurations for different AI providers are handled correctly.

**Claude configuration** tests ensure that Anthropic's official models work correctly with empty base URLs. This is important because Claude doesn't require a custom API endpoint.

**DeepSeek configuration** tests validate that the DeepSeek provider's specific base URL and model identifier are preserved correctly. DeepSeek uses a different API endpoint than Anthropic, so this must be configured properly.

**KIMI configuration** tests verify that the Moonshot AI provider's configuration is handled correctly, including its specific base URL and model naming conventions.

## Writing New CLI Integration Tests

As the CLI integration evolves, new tests may be needed to validate additional functionality. This section provides guidance on writing effective CLI integration tests.

### Test Structure

CLI integration tests follow the same basic structure as other tests but include additional setup and teardown for the temporary test environment:

```typescript
describe('Feature Name', () => {
  let testDir: string;
  let configFilePath: string;

  beforeEach(async () => {
    testDir = await fs.mkdtemp(path.join(os.tmpdir(), 'ccm-test-'));
    configFilePath = path.join(testDir, '.ccm_active_model');
  });

  afterEach(async () => {
    await fs.rm(testDir, { recursive: true, force: true });
  });

  it('should validate specific behavior', async () => {
    // Write test configuration
    await fs.writeFile(configFilePath, JSON.stringify(config));
    
    // Read and validate
    const content = await fs.readFile(configFilePath, 'utf-8');
    const parsed = JSON.parse(content);
    
    expect(parsed.provider).toBe('expected-value');
  });
});
```

This structure ensures each test has a clean environment and doesn't interfere with other tests.

### File Operation Best Practices

When writing tests that involve file operations, follow these best practices to ensure reliability.

**Always use async/await** for file operations rather than callbacks or synchronous methods. Async operations prevent blocking the test runner and make tests more reliable.

**Handle errors explicitly** by wrapping file operations in try-catch blocks or using expect assertions that validate error conditions. Don't assume file operations will succeed—test both success and failure paths.

**Clean up thoroughly** by removing all temporary files and directories in afterEach hooks. Use the `force: true` option with `fs.rm` to ensure cleanup succeeds even if files don't exist.

**Use absolute paths** when creating test files to avoid ambiguity about file locations. The `path.join` function ensures paths are constructed correctly across different operating systems.

### Testing Error Conditions

Error condition tests are particularly important for CLI integration because file operations can fail in many ways. When writing error tests, validate both that errors are thrown and that they contain useful information for debugging.

For example, when testing invalid JSON handling:

```typescript
it('should reject invalid JSON', async () => {
  await fs.writeFile(configFilePath, 'invalid{{{json');
  
  await expect(async () => {
    const content = await fs.readFile(configFilePath, 'utf-8');
    JSON.parse(content);
  }).rejects.toThrow();
});
```

This test verifies that parsing invalid JSON throws an error rather than returning undefined or causing a crash.

### Provider-Specific Test Data

When testing provider-specific configurations, use realistic data that matches actual provider requirements. Don't use generic placeholder values that might not catch real-world issues.

For example, a realistic DeepSeek configuration test uses the actual DeepSeek base URL and model identifier:

```typescript
const config = {
  provider: 'deepseek',
  timestamp: new Date().toISOString(),
  config: {
    ANTHROPIC_MODEL: 'deepseek-chat',
    ANTHROPIC_API_KEY: '$DEEPSEEK_API_KEY',
    ANTHROPIC_BASE_URL: 'https://api.deepseek.com/beta',
  },
};
```

Using realistic data ensures tests catch configuration errors that would occur in production.

## Integration with Dashboard

The CLI integration tests validate the file format and content produced by the dashboard, but they don't directly test the dashboard code. This separation is intentional and provides several benefits.

### Contract Testing Approach

The tests define a contract between the dashboard and CLI tools: the dashboard must produce files that match the expected format, and the CLI tools must correctly parse files in that format. As long as both sides honor this contract, they can evolve independently.

This contract-based approach allows the dashboard to be refactored or rewritten without changing the CLI tools, as long as it continues producing files in the expected format. Similarly, the CLI tools can be enhanced or optimized without requiring dashboard changes.

### Validation Points

The integration tests validate several critical aspects of the dashboard-CLI contract:

**File format** must be valid JSON with specific required fields. The tests verify that configurations include provider, timestamp, and config object fields with appropriate types.

**Field names** must match exactly between dashboard and CLI. The tests use the actual field names (`ANTHROPIC_MODEL`, `ANTHROPIC_API_KEY`, `ANTHROPIC_BASE_URL`) to ensure consistency.

**Value formats** must be compatible with shell environment variables. The tests verify that values are properly quoted and escaped for bash consumption.

**Provider identifiers** must match the supported provider list. The tests validate that provider names are consistent across the system.

## Continuous Integration

The CLI integration tests are designed to run in CI environments without modification. They don't depend on specific file system locations, user accounts, or network access.

### CI Configuration

A typical CI pipeline would run the complete test suite as part of the build process:

```yaml
test:
  script:
    - pnpm install
    - pnpm test
```

The test command exits with a non-zero status code if any tests fail, causing the CI build to fail and preventing broken code from being merged.

### Test Isolation

The temporary directory approach ensures tests are isolated from each other and from the CI environment. Tests don't interfere with other tests running in parallel, and they don't leave artifacts that could affect subsequent builds.

This isolation is critical for CI reliability because CI environments may run multiple builds concurrently or reuse build directories across builds.

## Debugging Failed Tests

When CLI integration tests fail, the error messages provide detailed information about what went wrong. This section explains how to interpret common failure patterns and diagnose issues.

### File Operation Failures

File operation failures typically indicate permission issues, missing directories, or file system errors. The error message will include the file path and operation that failed, making it easy to identify the problem.

For example, a write failure might produce an error like:

```
Error: ENOENT: no such file or directory, open '/nonexistent/path/.ccm_active_model'
```

This indicates that the parent directory doesn't exist. The fix is to ensure the directory is created before attempting to write the file.

### JSON Parsing Failures

JSON parsing failures indicate that the configuration file contains invalid JSON. The error message will include the position where parsing failed, helping identify the syntax error.

For example:

```
SyntaxError: Unexpected token { in JSON at position 8
```

This indicates that the JSON contains a syntax error at character position 8. Examining the file content around that position will reveal the problem.

### Assertion Failures

Assertion failures indicate that the actual value doesn't match the expected value. Vitest provides detailed output showing both values, making it easy to identify the discrepancy.

For example:

```
AssertionError: expected 'deepseek' to be 'claude'
Expected: "claude"
Received: "deepseek"
```

This indicates that the provider field contains 'deepseek' but the test expected 'claude'. This might indicate a test bug or a real issue with the configuration logic.

## Best Practices Summary

Effective CLI integration testing requires following established best practices that ensure tests are reliable, maintainable, and valuable.

**Use temporary directories** for all file operations to ensure test isolation and prevent interference between tests. Create a fresh directory for each test and clean it up afterward.

**Test realistic scenarios** using actual provider configurations and realistic data rather than generic placeholders. This ensures tests catch real-world issues.

**Validate error conditions** explicitly by testing that invalid inputs produce appropriate errors. Don't assume operations will succeed—test both success and failure paths.

**Keep tests focused** by validating one specific behavior per test. Complex tests that check multiple things are harder to understand and maintain.

**Use descriptive test names** that clearly explain what behavior is being validated. Good test names serve as documentation and make failures easier to understand.

**Clean up thoroughly** by removing all temporary files and directories in afterEach hooks. This prevents test artifacts from accumulating over time.
